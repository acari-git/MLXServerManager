# v0.2 Plan

v0.2 focuses on Setup Diagnostics for the existing Direct Mode server manager.

The goal is to help users confirm that their configured `mlx_lm.server` executable path, selected model profile, host, and port are usable before they start a managed server process.

## Scope

### Included

- Setup Diagnostics.
- Validation for the configured `mlx_lm.server executable path`.
- Validation for selected model host and port.
- Integration with the existing Port Check.
- Integration with the existing Ready Check through `GET /v1/models`.
- Display of the local configuration storage location.
- Diagnostics results in the UI and Logs.

### Candidate Only

- Minimal Model profile editing improvements.

Model profile editing may be considered if diagnostics need a small correction flow, but it is not the main v0.2 feature.

### Deferred

- Menu bar quick actions.
- LAN Web UI.
- App Intents.
- Auto unload.
- Proxy mode.
- Chat UI.
- Hugging Face download manager.
- Multiple simultaneous server management.

Menu bar quick actions should be considered for v0.3 or later.

## Product Constraints

v0.2 must preserve Direct Mode:

```text
OpenAI-compatible client -> mlx_lm.server
```

The app may inspect configuration, start and stop the managed process, check ports, and check readiness. It must not proxy inference traffic.

The app must not send `/v1/chat/completions`. Diagnostics must not run model inference.

## Diagnostics Goals

Setup Diagnostics should answer these questions:

- Is the `mlx_lm.server executable path` configured?
- Does the configured executable path exist?
- Is the configured path executable?
- Can the executable respond to `--help` or a similarly safe command?
- Are the selected host and port syntactically valid?
- Is the configured port available before Start?
- If the port is busy, is it likely occupied by an external process?
- Are required settings present before Start?
- After Start, does `GET /v1/models` return a Ready result?
- Where are `settings.json` and `models.json` stored?
- Are diagnostics results visible in Logs?

## UI Direction

Add either a Diagnostics panel or a `Run Diagnostics` button near the existing settings and status areas.

The UI should show a concise result for each diagnostic item:

- pass
- warning
- fail
- not run

Detailed explanations should go to Logs. SwiftUI views should render diagnostic state and send user intents only; validation and command execution should live in services.

## Safety Rules

- Keep Direct Mode.
- Do not add a proxy.
- Do not add Chat UI.
- Do not run `/v1/chat/completions` from the app.
- Do not run inference as part of diagnostics.
- Do not use `pkill`, `killall`, or `pgrep`.
- Do not stop externally launched `mlx_lm.server` processes.
- Do not hardcode user-specific absolute paths.
- Do not commit `settings.json`, `models.json`, model files, logs, `.env`, `HF_TOKEN`, or virtual environments.

## Expected Outcome

A user should be able to open the app, run diagnostics, and understand whether Start is expected to work before launching the server. If something is missing or unsafe, the app should identify the specific setting or environment issue without modifying external processes.
