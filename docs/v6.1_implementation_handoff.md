# v6.1 Implementation Handoff

## Status

- Added in `v6.0.5` as a docs-only handoff release.
- Prepares `v6.1.0` implementation work.
- No app-code changes are included in `v6.0.5`.
- No new app binary is included in `v6.0.5`.

## Purpose

`v6.1.0` should begin the Profiles / Model List Surface work on top of the completed `v6.0.x` App Shell foundation.

The goal is to introduce a staged Profiles destination without disrupting Dashboard v1, Direct Mode, runtime lifecycle behavior, import/export behavior, selected profile behavior, or current target behavior.

## v6.0.x Completed Foundation

- `v6.0.0`: App Shell / Sidebar Foundation.
- `v6.0.1`: App Shell Sidebar Polish.
- `v6.0.2`: App Shell Release Hygiene.
- `v6.0.3`: App Shell Stable Identifiers.
- `v6.0.4`: AppSection Metadata Tests.

## v6.1.0 Minimum Scope

1. Add `AppSection.profiles`.
2. Add stable Profiles metadata: id, title, subtitle, SF Symbol name, and accessibility identifier.
3. Add a lightweight Profiles destination view.
4. Mount Profiles through `AppShellView`.
5. Keep Dashboard as the default landing section.
6. Keep Dashboard model list and runtime controls unchanged.
7. Expand AppSection tests to cover Dashboard and Profiles.
8. Update README and task docs.

## Recommended Implementation Order

1. Update `AppSection` while preserving Dashboard raw value and metadata.
2. Add a lightweight Profiles surface.
3. Update the `ContentView` app-section switch.
4. Expand `AppSectionTests`.
5. Update docs.
6. Run build, tests, release build, zip verification, and GitHub Release checks.

## Dashboard Preservation Rules

Preserve:

- Dashboard as the default landing surface.
- Dashboard card order.
- Start / Stop / Restart behavior.
- Current Target behavior.
- readiness behavior.
- `/v1/models` readiness usage.
- Import / Export entry points.
- External Server Detection / Adopt / Forget behavior.

The Profiles surface must not become the primary runtime lifecycle surface in `v6.1.0`.

## Profiles Surface Guardrails

Allowed:

- top-level Profiles sidebar destination,
- profile/model list display,
- selected profile visibility,
- existing model metadata display,
- stable identifiers for future UI tests.

Avoid:

- rewriting profile persistence,
- changing selected profile semantics,
- changing current target semantics,
- moving Start / Stop / Restart ownership,
- changing import/export schema,
- adding model download, deletion, scanning, or cache cleanup,
- adding inference testing or Chat UI.

## Direct Mode Boundary

Direct Mode remains:

```text
OpenAI-compatible client -> mlx_lm.server or adopted external server -> MLX model
```

MLX Server Manager must not proxy inference requests, rewrite requests, inspect inference traffic, become a Chat UI, route between backends, own external server processes, or insert itself between clients and `mlx_lm.server`.

## Security / Privacy Boundary

`v6.1.0` must not add telemetry, analytics, crash reporting, request logging, request tracing, inference traffic inspection, API key management, token storage, secret persistence, generated client config persistence, or background monitoring.

## v6.1.0 Acceptance Criteria

- App sidebar includes Dashboard and Profiles.
- Dashboard remains the default selection.
- Dashboard v1 behavior is unchanged.
- Profiles surface renders without owning runtime lifecycle behavior.
- AppSection tests cover Dashboard and Profiles metadata.
- Build succeeds.
- Tests succeed.
- Release zip contains only the unsigned app bundle.
- No new inference proxy, Chat UI, request rewriting, request inspection, telemetry, or background monitoring behavior is introduced.

## Non-goals for v6.1.0

Do not include Detail Inspector, Logs Panel Refresh, Client Setup Surface, Metrics / System Context, Chat UI, model download, model deletion, installed model scanning, cache cleanup, telemetry, background monitoring, API key or token storage, proxy mode, multi-backend routing, request rewriting, or request inspection.
