# v0.6 Plan

v0.6 focuses on Model profile add and delete support for the existing Direct Mode server manager.

The goal is to let users maintain multiple saved model profiles in the UI while keeping runtime behavior safe, local, and limited to `models.json` profile data.

## Scope

### Included

- Add a new model profile from the UI.
- Delete an existing model profile from the UI.
- Persist added profiles to `models.json`.
- Persist deleted profile results to `models.json`.
- Select a newly added profile after save.
- Fall back safely to another profile when the selected profile is deleted.
- Keep at least one model profile at all times.
- Define duplicate `modelID` behavior.
- Define default values for:
  - `displayName`
  - `modelID`
  - `host`
  - `serverPort`
  - `enableThinking`
  - `notes`
- Require confirmation before deleting a profile.
- Clarify that deleting a profile does not delete model files, Hugging Face cache, or local directories.

### Deferred

- Multiple simultaneous server management.
- Running multiple models at the same time.
- Model file deletion.
- Hugging Face download manager.
- Model download.
- Automated model existence checks.
- Proxy mode.
- Chat UI.
- LAN Web UI.
- App Intents.
- Auto unload.
- CI/CD.
- Notarization.
- DMG creation.
- App Store distribution.

## Direct Mode

v0.6 must preserve Direct Mode:

```text
OpenAI-compatible client -> mlx_lm.server
```

Adding or deleting model profiles must not introduce a proxy, run inference, or send app-side `/v1/chat/completions` requests.

## UI Direction

Model profile management should extend the existing model list and model detail flow without a large layout rewrite.

Recommended UI placement:

- Add `Add Profile` near the model list.
- Add `Delete Profile` near the selected model detail or profile editor.
- Use the existing profile editor flow where practical.
- Require a confirmation dialog for delete.
- Explain when delete is blocked.

For `Add Profile`, v0.6 should prefer a predictable draft:

- Start from a blank draft with safe local defaults, or offer a future duplicate action separately.
- Use `127.0.0.1` for host.
- Use the app default port when available.
- Use `not-required-local` only for app-level API key placeholders, not as a model ID.
- Require a non-empty `modelID` before save.

Duplicating the selected profile can be considered later, but v0.6 should avoid accidentally creating indistinguishable duplicate profiles.

## Validation Rules

Add profile validation should match existing edit validation:

- `modelID` must not be empty.
- `host` must not be empty.
- `serverPort` must be an integer from 1 through 65535.
- Empty `displayName` should be filled with `modelID`.
- `notes` may be empty.
- `enableThinking` is a boolean.

Duplicate `modelID` behavior:

- v0.6 should block duplicate `modelID` values by default.
- The error should be shown in the UI and Logs.
- A future version may allow duplicates if profiles gain stable user-facing IDs.

## Delete Rules

Delete should only remove a profile entry from `models.json`.

It must not delete:

- Model files.
- Hugging Face cache.
- Local model directories.
- Logs.
- Runtime settings outside `models.json`.

Delete guard rules:

- The last remaining profile cannot be deleted.
- If a managed server is running, Delete Profile is blocked.
- If the selected profile is deleted while no managed process is running, select a safe fallback profile.

Fallback selection should be deterministic:

- Select the first remaining profile.

## Persistence

Model profile add and delete actions should save through the existing settings storage path for `models.json`.

Requirements:

- Save added profiles immediately after valid add.
- Save deleted profile list immediately after confirmed delete.
- Log save success or failure.
- Do not commit `models.json`.
- Do not include runtime settings or model files in Git.

## Running Process Behavior

Profile management must not change the currently running server implicitly.

While a managed process is running:

- Adding a profile is allowed if it does not alter the running profile.
- Deleting profiles is blocked.
- The UI should explain that the managed server must be stopped before deleting profiles.
- Stop still targets only the managed process held by the app.

Profile management must not start, stop, or restart `mlx_lm.server`.

## Safety Rules

- Keep Direct Mode.
- Do not add a proxy.
- Do not add Chat UI.
- Do not run `/v1/chat/completions` from the app.
- Do not run model inference when adding or deleting profiles.
- Do not launch `mlx_lm.server` from add or delete actions.
- Delete only profile data stored in `models.json`.
- Do not delete model files, Hugging Face cache, or local model directories.
- Stop only the managed process held by this app.
- Do not stop external processes.
- Do not use `pkill`, `killall`, or `pgrep`.
- Do not commit `settings.json` or `models.json`.
- Do not commit model files.
- Do not commit `.app` bundles or build artifacts.
- Do not hardcode user-specific absolute paths.
- Use placeholders such as `<model-id>` and `<path-to-model>` in examples.

## Expected Outcome

A user should be able to add, select, edit, and delete saved model profiles while preserving the existing single managed server model. The feature should make `models.json` easier to manage without touching model files or changing the inference path.
