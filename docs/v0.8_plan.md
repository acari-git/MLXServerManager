# v0.8 Plan

## Focus

v0.8 focuses on making Logs and Setup Diagnostics easier to use when something goes wrong.

The app now covers Start, Stop, Restart, Setup Diagnostics, profile editing, profile add/delete, and model switching. v0.8 should help users understand which operation failed, what was checked, and what to try next without changing the Direct Mode inference path.

Direct Mode remains unchanged:

```text
OpenAI-compatible client -> mlx_lm.server
```

The app must continue to manage `mlx_lm.server` directly. Log and diagnostics improvements must not introduce telemetry, cloud logging, inference requests, or a proxy.

## Goals

- Improve Logs readability.
- Make log category and severity easier to scan.
- Keep existing bounded log behavior.
- Keep Clear Logs.
- Improve copying logs for support or local troubleshooting.
- Improve Diagnostics result readability.
- Make Diagnostics summary easier to understand.
- Make warnings and failures easier to find.
- Organize Ready Check, Port Check, and executable path check results.
- Make Start, Stop, and Restart log sequences easier to follow.
- Make profile add/edit/delete/switch logs easier to follow.
- Add manual checklist guidance for what to inspect when troubleshooting.

## Log Categories

v0.8 should standardize and make visible the categories already implied by log lines:

- `info`
- `warning`
- `error`
- `start`
- `stop`
- `restart`
- `diagnostics`
- `profile`
- `switching`
- `ready`
- `port`
- `memory`
- `process`
- `settings`

The initial implementation can stay text based. Color or icon treatment is optional.

## Diagnostics Direction

Diagnostics should keep using the existing structured checks. v0.8 should improve presentation and summary clarity rather than adding risky new checks.

Diagnostics should emphasize:

- executable path configured
- executable path exists
- executable path is executable
- host and port validation
- Port Check result
- Ready Check result via `GET /v1/models`
- settings and models storage location
- failure and warning counts
- what to inspect next

## UI Direction

- Improve the existing `LogView` without a large rewrite.
- Keep text readable and bounded.
- Preserve Clear Logs.
- Consider `Copy Logs`.
- Consider `Copy Diagnostics Summary`.
- Keep filter/search optional.
- If filter/search adds complexity, defer it to a later version.
- Reuse existing Diagnostics UI.
- Avoid nested or heavy UI surfaces.

## Non-Goals

v0.8 does not add:

- Remote log sending.
- Telemetry.
- Crash reporting service.
- Analytics.
- External log collection service integration.
- Cloud logging.
- File-persistent logs.
- Automatic log upload.
- Proxy mode.
- Chat UI.
- LAN Web UI.
- App Intents.
- Auto unload.
- Hugging Face download manager.
- Model download.
- Model file deletion.
- Multiple simultaneous server management.
- CI/CD.
- Notarization.
- DMG creation.
- App Store distribution.

## Safety Constraints

- Keep Direct Mode.
- Do not add a proxy.
- Do not add Chat UI.
- Do not send `/v1/chat/completions` from the app.
- Log and diagnostics improvements must not run model inference.
- Log and diagnostics improvements must not start `mlx_lm.server` automatically.
- Diagnostics remain limited to safe checks and Ready Check through `/v1/models`.
- Stop must target only the managed process held by this app.
- External processes must not be stopped.
- `pkill`, `killall`, and `pgrep` must not be used.
- Model files, Hugging Face cache, and local model directories must not be deleted.
- `settings.json` and `models.json` must stay outside Git.
- Model files must stay outside Git.
- `.app` bundles and build artifacts must stay outside Git.
- Do not document personal fixed paths.
- Runtime logs may display user-selected local paths, but docs and Swift code must not hardcode personal paths.

## Expected Outcome

A user should be able to read Logs and Diagnostics results and quickly determine whether the next action is to fix the executable path, free a port, wait for readiness, stop a managed server, restart to apply a selected model, or inspect captured process output.
